Web Usability Considerations

Distinguishing link states

The standard specifies that each link must have a 3:1 contrasting ratio with the other colours' respective link states. The default (pristine) and hover are the salient states because of their prominence and, as such, had distinctive features in my design. In most cases, including mine, satisfying this constraint is difficult. For this reason, designers seldom adhere to the restriction. Instead, designers alternatively solely focus on changing appearance, independent of the colour. Not minding the colour requirements simplifies the design concerning accessibility without any compromises.

For text links, designers often choose the text-decoration-line CSS property to diversify the look under different states. In my case, I set underlines to appear on the hovered-over link text.

For images, since there is no text to decorate, I take an alternative approach. I enlarge the icon on hover to indicate the state change.

Responsive Navbar

My design conserves viewport space on smaller screens. If the viewport is below the medium screen size (from 768px to 991px in width), the navigation menu morphs into a hamburger menu. Such menus display the same information compactly, which is suitable for confined spaces. Such devices as mobile phones would profit from this consideration of responsiveness.

Familiar Imagry

The link to the landing page is an image of a home. The image introduces a home in the conceptual model. Implicitly, the image establishes a metaphor. The concept suggests that the landing page is where the user will likely spend the most time, similar to an actual home.

ARIA Annotations

Accessibility Rich Internet Applications (ARIA) detail additional information for screen readers. The information functions as hints on how screen readers should decipher the document's content.

ARIA Labels

Accepted Image accessibility guidelines suggest that ARIA labels should brief crucial contextual information about the functionality of websites that would otherwise be unusable to blind or visually impaired users. For instance, in my design, my home icon link informs screen readers about its purpose (the target resource), not its appearance.

Explicit Role Declarations

Describes the nature of the element to screen readers. My website explicitly mentions the "menu" and "menuitem" roles for the details disclosure element's content. The WAI-ARIA menu role denotes a transient interface component offering additional functionality to the user. The WAI-ARIA menuitem role role identify the individual controls associated with the menu.

HTML Section Heading Elements

As the name suggests, the purpose of HTML elements is to sectionize content. Appropriate usage of these tags is imperative to help screen readers fully comprehend the website's layout. To avoid any possibility of screen-reader misinterpretation, I designed each page to:

  • have only one level-one heading tag;
  • label sections sequentially;
  • be terse

HTML Sementic Tags

Using these tags is less error-prone as designers do not risk neglecting features promoting accessibility.

Details Disclosure Element

This element introduces a built-in implementation of an accordion. The enclosed <summary> tag implies button functionality out-of-the-box with the button role. This role identifies the element as keyboard-enabled.

The adjacent triangle provides affordance for the user, hinting the element has more content for sighted users.

Contrast

All present multimedia provides sufficient contrast with the background such that the content is legible.

According to WebAIM's colour contrast checker, the home icon's colour, #4242FF, is visible to all sighted people, including those with visual impairments.

Typography Settings

I used Bootstrap's default typography settings. However, to further enhance readability, I expanded the letter spacing to 2px in the navbar's brand section.
